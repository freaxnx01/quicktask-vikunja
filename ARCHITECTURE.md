# Architecture

## What this app does

QuickTask is an Android share-target app. The user shares a URL, text snippet, or file from any other Android app; QuickTask receives the share intent, optionally fetches the web page title for URLs, lets the user pick a Vikunja project, and creates the task via the Vikunja REST API. On Windows (desktop) there is no share intent; the app runs as a manual task-entry tool only.

---

## Entry points

**`lib/main.dart`** is the Flutter entry point. `runApp(QuickTaskApp())` mounts a single `MaterialApp` whose home is `HomePage`. `HomePage` owns two responsibilities: (1) listening to the share-intent stream from `receive_sharing_intent` and (2) routing between `SetupScreen`, `RecentTasksScreen`, and `ProjectPickerScreen` based on configuration state and incoming shares. The share listener is only started when `_isMobile` is `true` (`defaultTargetPlatform` is Android or iOS).

**`android/app/src/main/AndroidManifest.xml`** registers `MainActivity` as a share target. It declares `ACTION_SEND` intent filters for `text/plain` and `*/*` (single file), and `ACTION_SEND_MULTIPLE` for multiple files. The activity uses `launchMode="singleTop"` with `taskAffinity=""` to prevent a new task stack from being created on every incoming share.

## Module map

```
lib/
  main.dart                   App entry; share-intent listener; route orchestration
  build_info.dart             Generated by tool/build.sh — version + UTC timestamp
  models/                     Pure Dart data classes (no Flutter, no I/O)
    project.dart              Project (id, title, isArchived)
    vikunja_task.dart         CreateTaskRequest, TaskResponse, TaskSummary
    shared_content.dart       SharedContent (text?, files[]), SharedFile
  data/                       All I/O — widgets never call plugins directly
    secure_storage.dart       flutter_secure_storage wrapper; stores instance URL + API token
    vikunja_api.dart          HTTP client; GET projects, PUT task, PUT attachments
    vikunja_repository.dart   Orchestration over the API (pagination, task creation)
    title_fetcher.dart        URL detection + HTML <title> fetching → ResolvedTask
    share_intent_source.dart  Thin interface over receive_sharing_intent (injectable)
    task_history.dart         Recent 20 created tasks; JSON in shared_preferences
    project_usage_tracker.dart Last-used timestamp per project ID; shared_preferences
    project_favorites.dart    Set of favourite project IDs; shared_preferences
  ui/
    setup_screen.dart         URL + API token entry; validates via repository
    recent_tasks_screen.dart  Home screen; last 20 tasks; manual "Add task" entry
    project_picker_screen.dart Title editing + project list (recents, favourites, alpha)
    task_confirmation_screen.dart Success screen with task title, project, attachment status
    widgets/version_footer.dart   Small shared widget; displays app version
```

Every `lib/data/` class accepts its collaborators as optional constructor parameters so widget tests can inject hand-rolled fakes without a mocking framework.

## Data flow

A share from another Android app travels through the following steps:

1. Android OS matches the intent to `MainActivity` and delivers it to the `receive_sharing_intent` plugin.
2. `HomePage._handleShared()` receives `List<SharedMediaFile>` and converts it to `SharedContent` — text/URL in `text`, file paths in `files`.
3. If the app is not yet configured (no token stored), the shared content is held as `_pendingShared` and `SetupScreen` is shown. After a successful setup the pending content is forwarded.
4. If configured, `Navigator.pushAndRemoveUntil(ProjectPickerScreen, ...)` is called, clearing any stale route stack from a prior share.
5. `ProjectPickerScreen.initState()` calls `TitleFetcher.resolveTask(text, extraSubject)`:
   - Plain text → used as the task title directly.
   - URL with an `EXTRA_SUBJECT` supplied by the sharing app → that subject becomes the title; the URL is stored as the task description.
   - URL without a subject → HTTP GET with a 5 s timeout; `<title>` tag extracted from the HTML; raw URL used as fallback.
6. The user edits the title if needed and taps a project.
7. `VikunjaRepository.createTask(projectId, title, description: url?)` calls `PUT /api/v1/projects/{id}/tasks`.
8. If files were shared, `VikunjaRepository.uploadAttachments(taskId, filePaths)` sends them as a `multipart/form-data` PUT to `/api/v1/tasks/{id}/attachments`.
9. `TaskHistory.addEntry()` and `ProjectUsageTracker.recordUsage()` write to `shared_preferences`.
10. `TaskConfirmationScreen` is pushed. Tapping Done pops back to `RecentTasksScreen`.

## Auth

The Vikunja instance URL and API token are stored in `flutter_secure_storage` (Android Keystore / iOS Keychain / Windows Credential Manager). Every API request sends `Authorization: Bearer <token>`. The token is never logged, never shown in the UI in plaintext, and must not appear in error messages or PR descriptions. `SecureStorage.isConfigured` is the single gate checked before any API call.

## External dependencies

| Dependency | Purpose |
|---|---|
| Vikunja REST API | Task backend (`/api/v1/projects`, `/api/v1/projects/{id}/tasks`, `/api/v1/tasks/{id}/attachments`) |
| `http` | HTTP client; injected via constructor for testability |
| `html` | HTML parsing to extract `<title>` from fetched pages |
| `flutter_secure_storage` | OS-backed credential storage |
| `receive_sharing_intent` | Android/iOS share intent bridge; wrapped behind `ShareIntentSource` interface |
| `shared_preferences` | Non-sensitive local KV store (task history, project usage, favourites) |
| `provider` | Listed in `pubspec.yaml` but not wired yet |

## Build and run

See the `Makefile` for all targets. The key ones:

```bash
flutter pub get          # after checkout or pubspec changes
flutter run              # development build on a connected device
make apk                 # release APK → build/app/outputs/flutter-apk/app-release.apk
make windows             # release Windows build
flutter analyze          # must pass before commit
flutter test             # must pass before commit
```

Always use `make apk` / `make windows` for release builds — not `flutter build` directly. The `tool/build.sh` script stamps `lib/build_info.dart` with the current version and UTC timestamp before building; calling `flutter build` directly leaves that file stale.
